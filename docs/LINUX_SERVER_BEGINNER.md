# Linux и сервер: шаг за шагом (для начинающих)

Документ дополняется по мере работы. В каждом разделе — **один понятный способ** сделать задачу, без «десяти равноправных вариантов».

---

## Этап 1. Зачем это и как вообще попасть на сервер

### Зачем командная строка

На VPS (виртуальный сервер в облаке) часто нет «рабочего стола», как в Windows. Управляют им через **терминал**: вы вводите короткие **команды**, сервер отвечает текстом. Это нормально: так устроена большая часть серверов в интернете.

### Как вы оказываетесь «внутри» сервера

Обычно с **своего компьютера** подключаются по **SSH** — безопасное соединение с удалённой машиной.

**Один способ входа** (подставьте свои данные):

```bash
ssh root@ВАШ_IP
```

Например, если IP сервера `77.222.35.130`:

```bash
ssh root@77.222.35.130
```

Пароль (или ключ) вас спросит сам SSH при первом подключении. После успешного входа приглашение в терминале будет что-то вроде `root@имя-сервера:~#` — значит, вы уже **на сервере**, команды выполняются **там**, а не на домашнем ПК.

### Три команды, чтобы не потеряться

Все ниже вводятся **на сервере**, после входа по SSH.

- `pwd` ---> показать **текущую папку** (где вы сейчас находитесь).
- `ls` ---> показать **список файлов и папок** в текущей папке.
- `cd ПАПКА` ---> **перейти** в другую папку. Пример: `cd /srv/tg_mini_app`.

Чтобы подняться на уровень вверх (в родительскую папку):

- `cd ..` ---> выйти из текущей папки на один уровень вверх.

### Слово `sudo`

Многие системные действия (установка ПО, перезапуск служб) требуют прав администратора. Тогда перед командой пишут **`sudo`**:

```bash
sudo systemctl status nginx
```

Система может запросить **ваш пароль** (символы при вводе часто не отображаются — это нормально).

## Этап 2. Посмотреть и изменить файл на сервере

Всё по-прежнему **на сервере**, в SSH-сессии.

### Сначала просто посмотреть содержимое

- `cat ПУТЬ_К_ФАЙЛУ` ---> вывести **весь текст файла** в терминал (удобно для коротких файлов вроде настроек).

Пример:

```bash
cat /srv/tg_mini_app/.env
```

Если файл **очень длинный**, текст быстро «прокрутится» — тогда для просмотра лучше сразу открыть его в редакторе ниже.

### Редактирование: один редактор — **nano**

На типичном Ubuntu **nano** уже стоит. Один способ открыть файл для правки:

- `nano ПУТЬ_К_ФАЙЛУ` ---> открыть файл в простом редакторе (если файла не было, при сохранении он **создастся**).

Пример:

```bash
nano /srv/tg_mini_app/.env
```

**Как сохранить и выйти** (подсказки внизу экрана в nano; символ `^` значит клавиша **Ctrl**):

- **Ctrl+O** (буква О, не ноль) ---> записать изменения на диск; nano спросит имя файла — обычно достаточно нажать **Enter**.
- **Ctrl+X** ---> выйти из nano; если что-то меняли и не сохранили, спросит, сохранить ли — **Y** (да) или **N** (нет).

Пока вы внутри nano, обычные буквы **печатаются как в блокноте**. Стереть символ слева от курсора — **Backspace**.

### Небольшая осторожность

В файлах вроде **`.env`** лежат пароли и ключи: не отправляйте их скриншотами и текстом в чаты. Перед правкой можно сделать копию:

```bash
cp /srv/tg_mini_app/.env /srv/tg_mini_app/.env.bak
```

`cp откуда куда` ---> скопировать файл (если напортачите в `.env`, можно вернуться к копии).

## Этап 3. Загрузить файл на сервер и скачать с сервера

Команды **`scp`** вводятся **на вашем компьютере** (PowerShell, cmd или терминал с Linux/Mac) — **не** внутри SSH-сессии на сервере. Нужен установленный **OpenSSH** (часто уже есть в Windows 10/11 или вместе с Git for Windows).

При первом подключении спросит про ключ сервера — обычно пишут **`yes`** и Enter.

### Загрузить один файл **с ПК на сервер**

- `scp ПУТЬ_К_ФАЙЛУ_НА_ПК ПОЛЬЗОВАТЕЛЬ@IP:ПУТЬ_КУДА_НА_СЕРВЕРЕ` ---> **скопировать файл** на удалённую машину (папка на сервере должна уже существовать).

Пример (подставьте свой логин, IP и имя файла):

```bash
scp C:\Users\Вы\Desktop\notes.txt root@77.222.35.130:/srv/tg_mini_app/
```

Если в пути на Windows есть **пробелы** — возьмите путь в **кавычки**:

```bash
scp "C:\Users\Вы\Desktop\мои заметки.txt" root@77.222.35.130:/srv/tg_mini_app/
```

После команды введите **пароль SSH** для этого пользователя.

### Скачать один файл **с сервера на ПК**

- `scp ПОЛЬЗОВАТЕЛЬ@IP:ПУТЬ_НА_СЕРВЕРЕ ПУТЬ_КУДА_НА_ПК` ---> **скачать файл** с сервера в указанную папку (или под тем именем, которым закончится путь).

Пример: положить файл в текущую папку в PowerShell (где вы сейчас стоите командой `cd`):

```bash
scp root@77.222.35.130:/srv/tg_mini_app/README.md .
```

Точка **`.`** в конце значит «**сюда**, в текущую папку».

### Если нужна целая папка, а не один файл

Добавьте **`-r`** (рекурсивно) **перед** путями — только для копирования **папки** целиком:

- `scp -r ЛОКАЛЬНАЯ_ПАПКА ПОЛЬЗОВАТЕЛЬ@IP:ПУТЬ_НА_СЕРВЕРЕ` ---> загрузить папку на сервер.
- `scp -r ПОЛЬЗОВАТЕЛЬ@IP:ПУТЬ_К_ПАПКЕ_НА_СЕРВЕРЕ ЛОКАЛЬНАЯ_ПАПКА` ---> скачать папку с сервера.

## Этап 4. Какие на сервере службы и как проверить одну

Команды ниже вводятся **на сервере**, уже **после входа по SSH**.

На типичном VPS с Ubuntu сервисы управляются через **systemd** (имена часто заканчиваются на **`.service`**).

### Посмотреть, что **сейчас запущено**

- `systemctl list-units --type=service --state=running --no-pager` ---> вывести список **работающих** служб (список может быть длинным — пролистайте вверх в терминале или ищите глазами знакомые имена).

Флаг **`--no-pager`** нужен, чтобы весь текст сразу пошёл в окно терминала, а не открылся «листатель» внутри SSH.

### Проверить **одну** службу по имени

- `systemctl status ИМЯ.service` ---> показать **состояние** этой службы (работает / не работает / ошибка) и **несколько последних строк** из её журнала.

Пример:

```bash
sudo systemctl status nginx
```

Для **вашего** приложения по инструкции из репозитория чаще всего настроены **два** юнита (имена могут чуть отличаться, но обычно так):

- `tg-mini-app-api.service` — веб‑API (сайт, мини‑приложение).
- `tg-mini-app-bot.service` — процесс бота в Telegram.

Проверка:

```bash
sudo systemctl status tg-mini-app-api.service
sudo systemctl status tg-mini-app-bot.service
```

Строка **`Active: active (running)`** означает, что служба **запущена**. Если **`failed`** или **`inactive`** — по логам внизу вывода часто видно причину.

### Если **точное имя не помните**, но оно связано с проектом

- `systemctl list-units --type=service --all --no-pager | grep tg` ---> среди всех известных systemd служб показать только строки, где есть **`tg`** (часто всплывают `tg-mini-app-api` и `tg-mini-app-bot`).

Так же можно искать, например, **`nginx`**: замените в конце **`grep tg`** на **`grep nginx`**.

### Коротко перезапустить службу (когда правили код или `.env`)

Один типовой пример — после деплоя API:

```bash
sudo systemctl restart tg-mini-app-api.service
```

- `sudo systemctl restart ИМЯ.service` ---> остановить и снова **запустить** службу (применятся новые файлы и настройки, если они уже на диске).

## Этап 5. Логи службы подробнее (`journalctl`)

Команды **на сервере**, в SSH. Удобно, когда в **`systemctl status`** мало строк, а нужно увидеть **историю ошибок** (трейсбек Python, «порт занят», неверный `.env` и т.д.).

### Последние сообщения **одной** службы

- `sudo journalctl -u ИМЯ.service -n ЧИСЛО --no-pager` ---> показать **последние** столько-то **строк журнала** только этой службы (старые сверху, новые внизу в конце блока).

Пример — последние **80** строк API:

```bash
sudo journalctl -u tg-mini-app-api.service -n 80 --no-pager
```

Для бота замените имя на **`tg-mini-app-bot.service`**.

**`sudo`** нужен, чтобы точно видеть всё, что пишет система; **`--no-pager`** — чтобы текст сразу вывалился в терминал, как в этапе 4.

### Следить за логом **в реальном времени**

Полезно, когда вы **только что** нажали что-то в боте или открыли сайт и хотите увидеть реакцию в логе.

- `sudo journalctl -u ИМЯ.service -f` ---> **поток** новых строк (остановка — **Ctrl+C**).

Пример:

```bash
sudo journalctl -u tg-mini-app-api.service -f
```

Пока команда работает, терминал «занят» — это нормально.

## Этап 6. Nginx: где конфиг и как не сломать сайт

**Nginx** — программа, которая обычно принимает **HTTPS** с интернета и **проксирует** запросы на ваше приложение (например, на порт **8000**, где крутится Uvicorn). Команды **на сервере**, в SSH.

### Где обычно лежат настройки (Ubuntu)

- `ls /etc/nginx/sites-enabled/` ---> показать **подключённые** конфиги сайтов (по имени файла часто видно домен или `default`).

Править удобно файл в **`sites-available`** (там хранят оригинал), а в **`sites-enabled`** лежат **ссылки** на него — так принято в Debian/Ubuntu. Пример открыть в **nano** (этап 2):

```bash
sudo nano /etc/nginx/sites-available/default
```

Если у вас отдельный файл под домен — имя может быть вроде **`vikmybot.ru`**; его тоже ищите через **`ls`** в **`sites-available`**.

### Перед перезагрузкой — **обязательно проверить синтаксис**

- `sudo nginx -t` ---> проверить все конфиги: если написано **`syntax is ok`** и **`test is successful`**, можно применять; если **ошибка** — nginx подскажет файл и строку, **не делайте** `reload`, пока не исправите.

### Применить изменения после правки

- `sudo systemctl reload nginx` ---> перечитать конфиг **без** полного обрыва соединений (основной способ после удачного **`nginx -t`**).

Если **`reload`** не помогает при серьёзной поломке, одна запасная мера — полный перезапуск:

```bash
sudo systemctl restart nginx
```

### Если сайт «не открывается», а виноват ли nginx

- `sudo tail -n 50 /var/log/nginx/error.log` ---> последние строки **журнала ошибок** nginx (часто видно 502, SSL, «connection refused» к бэкенду).

## Этап 7. Проверить сайт и API с сервера (**curl**)

**Curl** — утилита «сходить по URL из терминала». Удобно, когда браузер на вашем ПК показывает ошибку, а вы хотите понять: **жив ли бэкенд** и **достучался ли nginx**. Команды вводите **на сервере** в SSH (как в этапе 4).

### Сначала — приложение **напрямую** (минуя nginx)

Так проверяют, что **Uvicorn** слушает порт (часто **8000** — смотрите в **`.env`** на сервере, переменная вроде порта приложения).

- `curl -sS http://127.0.0.1:8000/health` ---> запрос к **API** локально; в ответе обычно JSON вроде **`{"status":"ok"}`** (или похожее). Если **Connection refused** — служба API не запущена или другой порт.

Если нужен только **код ответа** (например, убедиться, что это **200**):

```bash
curl -sS -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8000/health
```

### Потом — как снаружи, через **HTTPS** и домен

Подставьте **свой** домен вместо примера.

- `curl -sS -o /dev/null -w "%{http_code}\n" https://vikmybot.ru/health` ---> через **nginx** и SSL; в консоли должна появиться строка **`200`**. Если **`000`** или долго «висит» — смотрите nginx (этап 6) и логи API (этап 5).

Страница мини-приложения в браузере обычно открывается как **`/webapp`** — её тоже можно проверить кодом:

```bash
curl -sS -o /dev/null -w "%{http_code}\n" https://vikmybot.ru/webapp
```

(Тот же принцип: ожидается **200**, если всё настроено.)

## Этап 8. Firewall на сервере (**ufw**)

**Firewall** — фильтр: какие **входящие** подключения к серверу разрешены с интернета. На Ubuntu чаще всего ставят **`ufw`** (Uncomplicated Firewall). Команды — **на сервере**, под **SSH**; часть шагов нужен **`sudo`**.

**Важно:** сначала разрешите **SSH**, и только потом включайте firewall — иначе можно **закрыть себе вход** и придётся заходить через консоль в панели хостинга.

### Разрешить нужные порты

- `sudo ufw allow OpenSSH` ---> разрешить **SSH** (обычно порт **22**), чтобы не потерять доступ после **`enable`**.

- `sudo ufw allow 80/tcp` ---> **HTTP** (часто нужен для редиректа на HTTPS и для обновления сертификатов).

- `sudo ufw allow 443/tcp` ---> **HTTPS** (сайт, мини-приложение, webhook’и снаружи).

### Посмотреть правила и включить

- `sudo ufw status verbose` ---> показать, что уже разрешено и включён ли firewall (**Status: active** или **inactive**).

- `sudo ufw enable` ---> **включить** firewall; спросит подтверждение — наберите **`y`** и Enter.

После смены правил при уже активном firewall:

- `sudo ufw reload` ---> применить обновлённые правила.

### Если снаружи всё равно «не достучаться»

Проверьте ещё **отдельный firewall / security group** в панели **VPS** (у провайдера) — он существует **поверх** ufw и тоже должен пропускать **80** и **443** (и **22** для SSH).
