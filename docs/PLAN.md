# tg_mini_app: план разработки (ТЗ и фактический код)

Документ связывает **исходное ТЗ заказчика** с **тем, как устроен код** после отладки и работы на сервере. Его можно дополнять по мере согласований. В списках ниже пояснения к пунктам даются через **`--->`**, как в **`docs/LINUX_SERVER_BEGINNER.md`**.

---

## Этап 1. Цель продукта

Telegram Mini App для заказа суши с UX, близким к агрегаторам доставки еды.

- **Бот** ---> согласование с оператором и выбор оплаты.
- **Веб-панели** ---> оператор (заказы, каталог) и курьер (доставка, наличные).
- **VPS** ---> развёртывание проверено (nginx, HTTPS, **systemd** для API и бота); пошаговые команды обновления и бэкапов — в **`docs/SERVER_RUNBOOK.md`**.

---

## Этап 2. Исходное ТЗ (краткая выжимка)

- **Шапка Mini App** ---> время работы; блок «Помощь», чтобы клиент мог сразу написать с вопросом.
- **Категории** ---> в т.ч. «Заказывали ранее», «Суши», «Роллы», «Сеты» (пример из переписки с заказчиком).
- **Каталог** ---> плитка с фото; кнопка «+» на карточке (внизу справа).
- **Корзина** ---> появляется при добавлении; у нас — **FAB** справа снизу.
- **Оформление** ---> адрес, время доставки; отправка на согласование.
- **Согласование** ---> оператору в бота: сводка и «Да» / «Нет» / «Изменить»; при «Изменить» — текст правок клиенту и «Да»/«Нет» у клиента.
- **Оплата** ---> после согласования «Наличные» / «Карта»; карта в ТЗ — **реальная** оплата через провайдера.
- **Веб-панель** ---> товары (фото, категория, название, вес, цена, описание, состав), категории, отметка выполнения заказов.
- **Дополнительно** ---> inline-заказ в чате; **общая корзина** (поделиться); команды в боте для управления.
- **Инфра** ---> сначала ПК, затем облако.

**Согласованное отклонение по категориям в сиде:** три раздела — «Горячие жаренные ролы», «Запеченые ролы», «Холодные ролы» — вместо детализации «Макидзуси / Хосомаки / Футомаки» из примера ТЗ. При необходимости структуру меню возвращаем к формулировкам заказчика.

## Этап 3. Текущее состояние по коду и продакшену

### 3.1. Сделано и работает

- **Backend (FastAPI)** ---> каталог **`/catalog/categories`**, **`/catalog/products`**; корзина **`/cart/*`**; заказы **`/orders/*`**, включая сценарий **замен позиций** (accept/reject substitutions); проверка **initData** Mini App; режим **`APP_ENV=local`** для отладки без Telegram; **`/health`**; диагностика **`/debug/last-order`** (на публичном проде лучше ограничить доступ).
- **БД (SQLite)** ---> категории, товары (в т.ч. **`stock_quantity`**), атрибуты и изображения, корзина с **`shared_key`**, заказы; правила статусов в **`order_flow.py`**.
- **Схема БД при старте** ---> **`create_all`** + **`db/schema_upgrade.py`** (добавление колонок к существующим таблицам). Пакет **alembic** перечислен в **`requirements.txt`**, но **ревизий миграций под проект нет** — для долгого продакшена разумно ввести явный процесс миграций.
- **Бот (aiogram)** ---> **`/start`**, кнопка меню / Web App, согласование (Да/Нет/Изменить), правки оператора и ответ клиента, выбор наличные/карта (**карта — заглушка**: нет Telegram Payments / **`PROVIDER_TOKEN`**), команды **`/ship`**, **`/delivery`**, **`/delivered`**, защита от двойного polling через lock-файл, настройка меню при старте.
- **Mini App** ---> **`webapp.html`** и **`app_v3.js`**: тема Telegram, вкладки категорий, сетка товаров, drawer корзины, оформление, отмена до согласования, опрос статуса, UI для **замен** позиций; ссылка **«Помощь»** открывает чат с оператором (**`OPERATOR_USERNAME`** → **`t.me/…`**, в Telegram WebView через **`openTelegramLink`**).
- **Панель оператора** **`/operator-panel`** ---> заказы, статусы, часть согласования как у бота, отметка оплаты, предложение и фиксация **замен** строк.
- **Редактор каталога** **`/operator-panel/catalog-manage`** ---> CRUD категорий и товаров, цена/вес/описание/атрибуты (состав), загрузка картинок в **`/catalog-media`**.
- **Курьер** **`/delivery/*`** при **`COURIER_API_TOKEN`** ---> список «в доставке», доставлено, учёт наличных (**`delivery_staff.py`**), статусы **`out_for_delivery`** / **`delivered`**.
- **API общей корзины** **`shared_key`**, **`GET /by-share/{shared_key}`**, **`GET /cart/{id}/share`** ---> **в Mini App нет** кнопки «поделиться» и привязки к **`start_param`**.

### 3.2. Частично или иначе, чем в ТЗ

- **Время работы в шапке** ---> по-прежнему текст в **`webapp.html`**; при необходимости вынести в настройки/API (см. фазу A).
- **Категории как в примере ТЗ** ---> три категории из сида; вкладки **«Заказывали ранее»** нет — нужны история заказов пользователя и API/UI.
- **Оплата картой** ---> заглушка в боте; для продакшена — Telegram Payments и обработка pre_checkout / successful_payment.
- **Inline-заказ в чате** ---> не реализовано.
- **Общая корзина для друзей** ---> backend готов, сценария в Mini App нет.
- **Облако** ---> реальный деплой на VPS и обновления через **`deploy/server-update.sh`** / **`push-and-deploy.cmd`** (детали в runbook).

## Этап 4. Дорожная карта (фазы A–E)

Ниже — логический порядок; часть задач можно вести параллельно.

### Фаза A — клиентское приложение (закрыть разрыв с ТЗ)

1. **«Помощь»** — сделано: **`OPERATOR_USERNAME`** в контексте **`/webapp`**, **`app_v3.js`** открывает **t.me** (в Mini App — **`openTelegramLink`**). Опционально: вынести **время работы** в настройки/API.
2. Согласовать финальный список **категорий**; при согласии — **«Заказывали ранее»** (API + вкладка).
3. Довести **карточки и FAB** до утверждённого с заказчиком вида.
4. **Поделиться корзиной** в Mini App: ссылка, **`start_param`**, присоединение к **`shared_key`**, правила одновременного редактирования.

### Фаза B — оплата и надёжность

1. Реальная **оплата картой** (invoice + провайдер Telegram).
2. По необходимости — журнал событий заказа.
3. **Управляемые миграции** вместо одного только **`schema_upgrade`**.

### Фаза C — веб-панель (доработки)

1. Улучшения редактора каталога (порядок, скрытие, массовые операции — по запросу).
2. Фильтры в списке заказов, роли «кухня / курьер» — если понадобится жёсткое разделение.

### Фаза D — расширенный Telegram

1. **Inline-режим** и открытие Mini App из чата.
2. Дополнительные **команды бота** под процессы заказчика.

### Фаза E — продакшен (поддержка)

1. Мониторинг, алерты, закрытие **`/debug/*`** от публичного интернета; **регулярные бэкапы** — скрипт **`deploy/backup-sqlite.sh`** и **cron** (см. **`docs/SERVER_RUNBOOK.md`**).

## Этап 5. Открытые вопросы к заказчику

- Финальные **категории** меню: остаёмся на трёх «роллах» или возвращаемся к «суши / роллы / сеты» и **«заказывали ранее»**?
- **«Помощь»** — только чат с оператором или ещё FAQ / телефон?
- **Платёжный провайдер** для Telegram (страна, валюта).
- **Общая корзина:** нужна одна корзина на N человек с одновременным редактированием или достаточно «одна ссылка — дополнили и всё»?
- **Роли** в панели: один оператор-админ или разделение кухня / курьер / офис?
