# Git и GitHub: шаг за шагом (для начинающих)

Документ дополняется по мере работы. В каждом разделе — **один понятный способ** сделать задачу, без «десяти равноправных вариантов».

**Git** — программа на вашем компьютере: версии файлов, история изменений, ветки. **GitHub** — сайт, где хранят копию репозитория и работают вместе (issue, pull request, обзоры кода). Обычно с ПК вы пользуетесь **Git**, а **GitHub** — это «удалённый» сервер с клоном проекта.

---

## Этап 1. Установить Git и представиться системе

### Где взять Git на Windows

Один удобный путь — **Git for Windows**: установщик ставит **git**, **bash** и часто уже включает **OpenSSH** (потом пригодится для ключей).

После установки откройте **Git Bash** или **PowerShell** и проверьте:

```bash
git --version
```

Должна появиться строка с номером версии.

### Имя и почта в коммитах

Git пишет автора в каждый коммит. Укажите **свои** данные (подставьте почту, с которой зарегистрированы на GitHub или которую не стыдно показать в истории):

```bash
git config --global user.name "Ваше Имя"
git config --global user.email "you@example.com"
```

Проверить:

```bash
git config --global --list
```

## Этап 2. Аккаунт на GitHub и доступ к репозиторию

### Регистрация

На [https://github.com](https://github.com) создайте аккаунт. **Пароли и токены** в чаты и скриншоты не скидывайте.

### Два способа входа Git к GitHub (выберите один и придерживайтесь его)

**Вариант А — HTTPS** (проще начать): при `git push` / `git pull` GitHub попросит логин. Вместо пароля от аккаунта с 2021 года обычно нужен **Personal Access Token** (настройки GitHub: Developer settings → Personal access tokens). Токен один раз создаёте и храните как пароль (менеджер паролей, не в репозитории).

**Вариант B — SSH** (удобно, когда освоились): на ПК генерируете пару ключей, **публичный** ключ добавляете в GitHub (Settings → SSH and GPG keys). Тогда URL репозитория будет вида `git@github.com:USER/REPO.git`, а пароль при каждом push не спрашивают (если ключ настроен).

В этом файле дальше в примерах будет **HTTPS**-URL; если перешли на SSH — замените URL на SSH из кнопки **Code** на странице репозитория.

### «Свой» репозиторий и «чужой»

- **Свой** (вы создали или вас добавили collaborator) — можно пушить в основную ветку или по правилам команды.
- **Чужой** (открытый проект) — **не пушат** напрямую: делают **fork**, потом **pull request**. Этап 8 ниже — кратко про это.

## Этап 3. Скачать репозиторий на компьютер (`clone`)

Рабочая папка с `.git` называется **локальным репозиторием**. Один способ получить её с GitHub:

- **`git clone URL`** ---> скачать копию и сразу связать с удалённым репозиторием (`origin`).

На странице репозитория на GitHub: зелёная кнопка **Code** → скопируйте HTTPS-URL. Пример:

```bash
cd C:\Users\Вы\Projects
git clone https://github.com/USER/имя-репозитория.git
```

Появится папка `имя-репозитория` с файлами проекта. Дальше **все команды Git** выполняйте **из этой папки** (или из подпапок того же репозитория):

```bash
cd имя-репозитория
```

## Этап 4. Ежедневный цикл: статус, сохранить, отправить на GitHub

Все команды ниже — **внутри папки клонированного репозитория** (терминал: PowerShell, cmd или Git Bash).

### Что изменилось

- **`git status`** ---> показать **изменённые** и **новые** файлы, что уже в индексе, что нет.

### Добавить файлы в «снимок» перед коммитом

- **`git add ПУТЬ_К_ФАЙЛУ`** ---> подготовить **конкретный** файл к коммиту.
- **`git add .`** ---> подготовить **все** изменения в текущей папке и ниже (удобно, но смотрите **`status`**, чтобы не закоммитить лишнее).

**Не коммитьте** секреты: пароли, ключи API, личные `.env` — в репозиторий им не место. Обычно в проекте есть **`.gitignore`**; если добавили секрет и уже закоммитили — одной отменой не отмоешься, нужно **ротация ключей** и очистка истории (отдельная тема). Проще заранее не класть секреты в tracked files.

### Зафиксировать версию локально

- **`git commit -m "Коротко, что сделали"`** ---> создать **коммит** с текущим индексом. Сообщение пишите осмысленно: потом по истории проще искать.

Пример цепочки:

```bash
git status
git add .
git commit -m "Исправить расчёт таймаута в боте"
```

Если Git жалуется, что «nothing to commit» — либо нет изменений, либо вы забыли **`git add`**.

### Отправить коммиты на GitHub

- **`git push`** ---> отправить ваши **новые коммиты** в **ту ветку**, где вы стоите, на удалённый репозиторий (обычно **`origin`**).

Первый раз для новой ветки Git может подсказать длинную команду вроде **`git push --set-upstream origin имя-ветки`** — скопируйте и выполните её один раз.

### Забрать чужие изменения с GitHub

Перед работой или перед push полезно обновиться:

- **`git pull`** ---> **скачать** изменения с **`origin`** и **влить** в текущую ветку (если не настроено иное — это часто `git fetch` + `git merge`).

Если **`pull`** ругается на **конфликт** — в файлах появятся маркеры `<<<<<<<` / `=======` / `>>>>>>>`. Их нужно **вручную** поправить, затем **`git add`** конфликтные файлы и **`git commit`** (или завершить merge, как подскажет Git).

## Этап 5. Ветки: одна фича — одна ветка

**Ветка** — указатель на коммит; **`main`** (или **`master`**) часто считают основной стабильной линией.

### Посмотреть ветки

- **`git branch`** ---> локальные ветки; звёздочка **`*`** — где вы сейчас.
- **`git branch -a`** ---> локальные и **удалённые** (с префиксом **`remotes/origin/`**).

### Создать ветку и переключиться

Один привычный способ:

```bash
git checkout -b feature/краткое-имя
```

Теперь коммиты идут в **`feature/краткое-имя`**, пока вы не переключитесь снова.

### Вернуться на main и обновиться

```bash
git checkout main
git pull
```

### Отправить новую ветку на GitHub

```bash
git push -u origin feature/краткое-имя
```

Дальше на GitHub можно открыть **Pull Request**: «влить ветку в **main**».

## Этап 6. Посмотреть историю и отличия

- **`git log --oneline -n 20`** ---> последние **20** коммитов одной строкой на коммит.
- **`git diff`** ---> что изменено в **рабочей папке**, но ещё **не** в индексе.
- **`git diff --staged`** ---> что уже в индексе (после **`add`**) и попадёт в следующий коммит.

## Этап 7. Откатить простые случаи (осторожно)

### Файл испортили, коммит ещё не делали

Вернуть файл к последнему закоммиченному виду (**потеряете несохранённые правки в этом файле**):

```bash
git checkout -- ПУТЬ/К/ФАЙЛУ
```

В новых версиях Git то же делают через **`git restore ПУТЬ/К/ФАЙЛУ`** — если команда есть у вас, можно её.

### Последний коммит только что сделали — исправить сообщение

```bash
git commit --amend -m "Новое сообщение"
```

Если **уже успели `push`** — в простых командах лучше не «переписывать» историю без нужды; для командной работы иногда проще **новый коммит** с исправлением.

### «Отменить» уже отправленный коммит аккуратно

Для начинающих безопаснее не **`revert`**/`reset` по памяти, а спросить в команде или прочитать runbook: **жёсткий reset** и **force push** ломают чужую историю.

## Этап 8. Fork и Pull Request (чужой открытый репозиторий)

1. На GitHub нажимаете **Fork** — появляется копия репозитория **в вашем аккаунте**.
2. **`git clone`** с **вашего** fork.
3. Делаете ветку, правки, **`commit`**, **`git push`** в **ваш** fork.
4. На GitHub в **вашем** fork — **Contribute** / **Open pull request** → выбор **upstream** (оригинальный проект) и ветки для слияния.

Мейнтейнеры смотрят PR, могут попросить правки; вы добавляете коммиты в **ту же ветку** — PR обновится сам.

## Этап 9. Полезные привычки и частые ошибки

- Перед **`add`** смотрите **`git status`** и **`diff`** — не закоммить случайно огромный файл или ключ.
- **`.env`** с секретами держите **вне** Git или только как **`.env.example`** без реальных значений.
- Если **`push` отклонён** (non-fast-forward) — сначала **`git pull`** (или **`git pull --rebase`** по правилам команды), разрешите конфликты, потом снова **`push`**.
- Если Git пишет про **`LF`/`CRLF`** — на Windows это нормальная тема; часто помогает настройка **`core.autocrlf`** по документации Git for Windows; для команды лучше следовать одному соглашению.
