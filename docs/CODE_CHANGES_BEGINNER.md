# Правки в коде и выкладка: шаг за шагом (для начинающих)

Документ дополняется по мере работы. В каждом разделе — **один понятный способ** пройти цикл «изменил код → проверил → сохранил в Git → обновил сервер», без лишних ветвлений.

Ниже **команды для терминала**. На **Windows** удобны **PowerShell** или **Git Bash**; на **сервере (Linux)** — оболочка **bash** после входа по **SSH**. Подставляйте свои пути, IP и имена remotes.

Детали по серверу, nginx, systemd и резервным копиям — в **`docs/SERVER_RUNBOOK.md`**. Базовые команды Linux — в **`docs/LINUX_SERVER_BEGINNER.md`**. Git и GitHub с нуля — в **`docs/GITHUB_BEGINNER.md`**.

## Этап 1. Локально: убедиться, что проект в порядке

Делайте **на своём ПК**, в **корне репозитория** (папка, где лежит **`pyproject.toml`** / **`src`**).

### Перейти в папку проекта (Windows, PowerShell)

```powershell
cd C:\tg_mini_app
```

(Если проект у вас в другом месте — замените путь.)

### Короткая проверка тестов

Один способ (из корня репозитория): виртуальное окружение активировано, выполнены **`pip install -r requirements.txt`** и **`pip install -e .`** (тогда пакет **`tg_mini_app`** уже в пути интерпретатора).

```powershell
python -m pytest tests -q
```

- **`pytest`** входит в **`requirements.txt`**; **`-q`** — менее подробный вывод.

Если по каким-то причинам **не** ставили проект командой **`pip install -e .`**, можно один раз для сессии задать путь к исходникам:

```powershell
$env:PYTHONPATH = "$PWD\src"
python -m pytest tests -q
```

Если **`python`** не находится, попробуйте **`py -3.12`** или **`py -3.13`** вместо **`python`** — как у вас установлено.

Опционально (если ставили **ruff** / **mypy**):

```powershell
python -m ruff check src tests
python -m mypy src/tg_mini_app
```

### Не закоммитить секреты

Перед следующим этапом откройте **`git status`** и убедитесь среди файлов **нет** **`data/app.db`**, **`.env`** с паролями и прочих вещей из **`.gitignore`**.

```powershell
git status
```

## Этап 2. Сохранить изменения в Git и отправить на GitHub

Команды **на ПК**, в корне репозитория.

### Посмотреть, что изменилось

- **`git status`** ---> список изменённых и новых файлов.

### Добавить файлы и сделать коммит

Один типовой вариант — всё релевантное сразу:

```powershell
git add .
git commit -m "Кратко: что сделали (например: каталог API и витрина)"
```

Сообщение коммита должно **по смыслу** объяснять правку — через месяц проще искать в истории.

### Подтянуть чужие коммиты (если репозиторий общий)

Если **`git push`** потом ругается, что нужно сначала **`pull`**, один привычный порядок:

```powershell
git pull --rebase
```

При **конфликтах** Git укажет файлы — правите вручную, затем **`git add`** этих файлов и снова **`git rebase --continue`** (как подскажет Git). Подробнее — **`docs/GITHUB_BEGINNER.md`**.

### Отправить на GitHub

```powershell
git push
```

Если основной remote у вас называется не **`origin`**, а например **`github`**, смотрите список:

- **`git remote -v`** ---> как называется remote и его URL.

Тогда push может быть таким:

```powershell
git push github main
```

(Имя ветки вместо **`main`** подставьте своё, если отличается.)

## Этап 3. Обновить код на сервере и перезапустить службы

Команды ниже относятся к **VPS с Linux**, где уже крутится проект (см. **`SERVER_RUNBOOK.md`**). Сначала зайдите по **SSH** (как в **`LINUX_SERVER_BEGINNER.md`**, этап 1).

### Резервная копия перед обновлением

Перед **`git pull`** на сервере имеет смысл сделать копию **`.env`** и файла БД (часто **`./data/app.db`** или путь из **`DATABASE_URL`**). Конкретные команды — в **`SERVER_RUNBOOK.md`** (раздел про резервную копию на VPS). Не пропускайте это, если боитесь потерять данные.

### Вариант А — одной командой с ПК (если настроили скрипт)

Из корня проекта на **Windows**, если у вас есть **`push-and-deploy.cmd`** и настроены **SSH** и **Git** (см. **`SERVER_RUNBOOK.md`**, раздел **8.5**):

```text
push-and-deploy.cmd -SshTarget root@ВАШ_IP
```

Подставьте **логин@сервер** как вы реально заходите по SSH. Если нужен другой remote или ветка — смотрите параметры в **runbook**.

### Вариант B — вручную на сервере

После **`ssh`** на сервере:

```bash
cd /srv/tg_mini_app
git pull
```

- Путь **`/srv/tg_mini_app`** замените на **свой**, если проект лежит иначе (смотрите **`pwd`** и **`ls`** на сервере).

Дальше обновить зависимости из **venv** (если у вас так принято в runbook; путь к **`activate`** может отличаться):

```bash
source /srv/tg_mini_app/.venv/bin/activate
pip install -e .
```

Или запустить готовый скрипт обновления, если он у вас есть:

```bash
bash /srv/tg_mini_app/deploy/server-update.sh
```

(Точная роль скрипта — в **`SERVER_RUNBOOK.md`**: обычно **pull**, **pip**, **restart** служб.)

### Перезапуск API и бота (systemd)

Один типовой набор имён из документации проекта:

```bash
sudo systemctl restart tg-mini-app-api.service tg-mini-app-bot.service
```

- **`sudo systemctl restart …`** ---> остановить и снова запустить службы, чтобы подхватились новый код и переменные окружения.

Проверка:

```bash
sudo systemctl status tg-mini-app-api.service
sudo systemctl status tg-mini-app-bot.service
```

Строка **`Active: active (running)`** означает, что служба **запущена**. Если **`failed`** — смотрите логи (**`SERVER_RUNBOOK.md`**, **`journalctl`**).

**Важно:** не держите второй ручной запуск **uvicorn** в SSH, если тот же порт уже занят **systemd** — будет **«address already in use»**. Сначала **остановите** службу для отладки или не запускайте дубль (**`SERVER_RUNBOOK.md`**).

## Этап 4. Быстрая проверка после выкладки

### Снаружи: здоровье и витрина

С **ПК** или с **сервера** (на сервере удобно **`curl`**):

- Запрос к **`/health`** — в ответе обычно **`"status":"ok"`**.
- Страница мини-приложения и каталог — как у вас принято (**`/webapp`**, **`/catalog/products`**).

Пример с сервера (подставьте свой домен или **`127.0.0.1:8000`**):

```bash
curl -sS -o /dev/null -w "%{http_code}\n" https://ВАШ_ДОМЕН/health
```

Код **`200`** — страница ответила успешно.

### Если правили модели БД

При старте API проект обычно создаёт недостающие **таблицы** и догоняет **простые** изменения схемы (добавление колонок) через свой механизм. Если что-то пошло не так — смотрите лог **`tg-mini-app-api`** (**`journalctl`**) и убедитесь, что **`DATABASE_URL`** на сервере указывает на нужный файл.

### Nginx

Если меняли только backend, перезапуск **nginx** чаще **не** нужен. Если трогали конфиг — **`sudo nginx -t`**, затем **`sudo systemctl reload nginx`** (как в **`LINUX_SERVER_BEGINNER.md`**, этап 6).
